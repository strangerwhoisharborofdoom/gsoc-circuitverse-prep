# CircuitVerse Architecture Analysis

## Overview

CircuitVerse is a free, open-source digital logic circuit simulator built as a full-stack web application. The platform consists of three main layers:

1. **Frontend** - jQuery/Vanilla JavaScript simulator (legacy) + Vue.js simulator (in development)
2. **Backend** - Ruby on Rails 8
3. **Database** - PostgreSQL with ActiveRecord

## Tech Stack Breakdown

### Backend (Ruby on Rails)

- **Framework**: Ruby on Rails 8
- **Languages**: Ruby (28.8% of codebase), JavaScript (41.3%), HTML (22.7%)
- **Database**: PostgreSQL with pg_search for full-text search
- **Authentication**: Devise-based authentication system
- **API**: JSON:API compliant REST API at `/api/v1/`

Key backend directories:
- `app/controllers/` - Rails controllers including API v1 endpoints
- `app/models/` - ActiveRecord models (Project, User, Tag, Star, etc.)
- `app/views/` - ERB templates for server-rendered HTML
- `app/assets/` - Static assets (CSS, JS, images)
- `config/` - Rails configuration files
- `apidoc/v1/` - API documentation

### Frontend (Legacy Simulator)

- **Core**: jQuery 3.x + Vanilla JavaScript
- **Canvas Rendering**: HTML5 Canvas API for circuit visualization
- **UI Components**: Bootstrap 5, jQuery UI
- **State Management**: Global scope-based state (no modern state management)
- **Key Files**:
  - `public/js/data.js` - Circuit save/load logic (~1000 lines)
  - `public/js/` - Simulator engine code

### Frontend (Vue Simulator - In Development)

- **Framework**: Vue.js 3 with TypeScript
- **State Management**: Pinia
- **Build Tool**: Vite
- **Location**: `cv-frontend-vue` subdirectory (git submodule)
- **Status**: Active development, not yet default in production

## Circuit Data Storage & Loading

### Data Model: Project -> ProjectDatum

The circuit data follows a parent-child model:

```
Project (PostgreSQL table)
  - id, name, description, slug, author_id
  - project_access_type (Public/Private)
  - view count, forked_project_id
  - image_preview (CarrierWave uploader)
  - has_one :project_datum

ProjectDatum (PostgreSQL table)
  - id, project_id (foreign key)
  - data (JSONB column - contains full circuit data)
```

The `ProjectDatum.data` column stores the **entire circuit as a JSON string** in the database[web:35][web:39][screenshot:37].

### Circuit JSON Format

The circuit data JSON structure (generated by `generateSaveData()` in `data.js`) contains:

```json
{
  "name": "Project Name",
  "projectId": "randomId",
  "timePeriod": 500,
  "clockEnabled": true,
  "focussedCircuit": "circuitId",
  "scopes": [
    {
      "id": "circuitId",
      "name": "main",
      "layout": { ... },
      "allNodes": [ ... ],
      "Input": [ ... ],
      "Output": [ ... ],
      "AND": [ ... ],
      "OR": [ ... ],
      "NOT": [ ... ],
      "NAND": [ ... ],
      "NOR": [ ... ],
      "XOR": [ ... ],
      "XNOR": [ ... ],
      "Multiplexer": [ ... ],
      "Splitter": [ ... ],
      "TriState": [ ... ],
      "Buffer": [ ... ],
      "ControlledInverter": [ ... ],
      "Clock": [ ... ],
      "ConstantVal": [ ... ],
      "Ground": [ ... ],
      "Power": [ ... ],
      "Input": [ ... ],
      "Output": [ ... ],
      "LED": [ ... ],
      "Button": [ ... ],
      "SevenSegDisplay": [ ... ],
      "HexDisplay": [ ... ],
      "Stepper": [ ... ],
      "SubCircuit": [ ... ],
      "restrictedCircuitElementsUsed": [],
      "nodes": []
    }
  ]
}
```

Each scope (circuit) contains:
- **Layout info**: Positioning and dimensions
- **allNodes**: All connection points in the circuit
- **Module arrays**: Lists of each component type (AND gates, OR gates, etc.)
- **restrictedCircuitElementsUsed**: Elements with special restrictions

### Save Flow (Frontend to Backend)

1. User clicks Save in simulator
2. `save()` function in `data.js` calls `generateSaveData()`[web:43]
3. Data is serialized to JSON string
4. AJAX POST to `/simulator/update_data` (or form POST to `/simulator/create_data`)
5. `ProjectsController#update_circuit` or `#create` handles the request[web:35]
6. `build_project_datum()` stores JSON in `ProjectDatum.data` column
7. Image preview is also saved via CarrierWave

### Load Flow (Backend to Frontend)

1. User opens a project URL
2. Backend renders project view with circuit data embedded
3. `GET /api/v1/projects/:id/circuit_data` returns raw JSON data[web:17]
4. Frontend `load()` function in `data.js` parses the JSON
5. `loadScope()` iterates through scopes and calls `loadModule()` for each component
6. Components are instantiated via `new window[objectType]()`
7. Connections are reconstructed from node data

### Key API Endpoints

| Method | Endpoint | Purpose |
|--------|----------|----------|
| GET | `/api/v1/projects` | List all public projects |
| GET | `/api/v1/projects/:id` | Get project details |
| GET | `/api/v1/projects/:id/circuit_data` | Get raw circuit JSON |
| POST | `/api/v1/projects` | Create new project |
| PATCH | `/api/v1/projects/:id` | Update project metadata |
| PATCH | `/api/v1/projects/update_circuit` | Save circuit data |
| DELETE | `/api/v1/projects/:id` | Delete project |
| GET | `/api/v1/projects/featured` | Get featured circuits |
| GET | `/api/v1/projects/search?q=` | Search projects |
| POST | `/api/v1/projects/:id/fork` | Fork a project |
| GET | `/api/v1/projects/:id/toggle-star` | Star/unstar |

## Simulator Component Architecture

### Module System

CircuitVerse uses a **module-based component system** where each logic element is a class:

- All components inherit from a base `CircuitElement` class
- Each component has:
  - Position (x, y coordinates)
  - Direction (orientation on canvas)
  - Label and labelDirection
  - Propagation delay
  - Input/output nodes
  - Custom properties (bit width, etc.)
  - `saveObject()` method to serialize state
  - `draw()` method to render on canvas

### Scope System

- Each project can have multiple **scopes** (circuits)
- Scopes are stored in `scopeList` object keyed by ID
- One scope is designated as `globalScope` (currently active)
- Scopes can depend on each other via SubCircuit references
- Dependency order is calculated for proper save/load sequence

### Rendering Pipeline

1. **updateOrder**: Simulation update sequence (topological sort)
2. **renderOrder**: Canvas drawing sequence
3. Canvas is cleared and redrawn on each simulation tick
4. SVG export is supported via canvas2svg library

## Extension Points for Learning Workflow

Based on the architecture analysis, the following integration points are ideal for the learning workflow features:

### 1. Template Library Integration

- **Storage**: New `Template` model (name, description, tags, difficulty, circuit_data JSONB)
- **API**: `GET /api/v1/templates`, `GET /api/v1/templates/:id/circuit_data`
- **Frontend**: Template browser sidebar in simulator, import button

### 2. Tutorial System Integration

- **Storage**: New `Tutorial` model with steps containing expected circuit states
- **Frontend**: Tutorial panel alongside simulator with step-by-step instructions
- **Validation**: Compare user's circuit data JSON against expected step output

### 3. Circuit Comparison

- Use `generateSaveData()` output to compare user circuits against template/tutorial targets
- Normalize component positions to allow flexible layout
- Verify logical correctness via simulation output comparison

### 4. Frontend Integration Strategy

For the **legacy jQuery simulator**:
- Add new panel/sidebar using existing DOM structure
- Use jQuery for DOM manipulation
- Integrate with existing `load()` function for template import

For the **Vue simulator**:
- Create Vue components for template browser and tutorial panel
- Use Pinia stores for tutorial state management
- Integrate with Vue simulator's loading mechanisms

### Usage in Learning Workflow

These templates serve as:

1. **Starting Points**: Students can import templates as base circuits to modify and experiment with
2. **Reference Designs**: Correct implementations of standard digital logic circuits
3. **Tutorial Examples**: Each template includes step-by-step construction instructions in the circuit comments/metadata
4. **Validation Targets**: Tutorial systems can compare student circuits against template outputs

### Integration with CircuitVerse

Templates can be integrated into CircuitVerse via:

- **API**: `GET /api/v1/templates` - List available templates
- **API**: `GET /api/v1/templates/:id/circuit_data` - Fetch specific template data
- **Frontend**: Template browser panel in the simulator sidebar
- **Import**: Use existing `load()` function in `data.js` to load template JSON directly
